# 포인트 조회 + 추천인 + 관리자 포인트 정책 — 구현 절차

> 대상 범위: PRD 9.3(포인트 정책) / 9.4(포인트 이벤트) / 9.5(추천인 코드)
> 스택: Next.js 16, React 19, Supabase(Postgres + Auth + RLS)

---

## 0. 핵심 개념 먼저

이 서비스의 포인트는 **"잔액 컬럼"을 두지 않는다.** 대신 지급/차감을 하나하나 `point_events`
행으로 쌓고, **총 포인트 = 이벤트 합계(SUM)** 로 계산한다. (회계의 원장/ledger 방식)

- 장점: 언제 왜 얼마가 바뀌었는지 100% 추적 가능, 정합성 깨질 일이 없음
- 지급 수치는 **코드에 하드코딩하지 않고** `point_policies` 테이블에서 읽는다 → 관리자가 값만 바꾸면 즉시 반영
- 정책이 `is_active = false`면 지급하지 않는다

3개 개념이 맞물린다:

| 개념 | 테이블 | 역할 |
| --- | --- | --- |
| 포인트 정책 | `point_policies` | "어떤 상황에 몇 점 줄지"의 규칙 (관리자 수정) |
| 포인트 이벤트 | `point_events` | 실제 지급/차감 기록 (합계가 곧 잔액) |
| 추천인 | `referral_codes`, `referrals` | 누가 누구를 초대했는지, 중복 방지 |

## 1. 현재 구현 상태 (이미 되어 있는 것)

작업 전에 반드시 아는 것부터: 이 3개 기능의 **뼈대는 이미 존재**한다. 아래는 "이미 됨" 목록이다.

- DB 스키마 5종: `point_policies` / `point_events` / `referral_codes` / `referrals` / `profiles`
  → `supabase/setup_all.sql`
- 정책 시드 5종: `signup(1000)`, `referral_join(2000)`, `referral_inviter(3000)`,
  `consultation_reserved(500)`, `payment_completed(5000)`
- 지급 코어 로직: `lib/points.ts`
  - `sumPoints(events)` — 합계 계산
  - `grantPoints(db, {userId, policyKey, ...})` — 정책 읽고 활성일 때만 적립, 비활성/미존재면 `null`
- 가입 후처리(멱등): `lib/after-signup.ts` → 신규가입 지급 + 추천 처리 + 1/3/7일 메시지 예약
- 포인트 조회 화면: `app/points/page.tsx`
- 추천 코드 화면: `app/referral/page.tsx`
- 관리자 정책 수정 API: `app/api/admin/policies/route.ts` (PATCH, admin 가드)
- 관리자 정책 화면: `app/admin/policies/page.tsx` + `PolicyRow.tsx`
- 테스트: `lib/points.test.ts`, `lib/after-signup.test.ts`

즉, 이 문서의 "절차"는 대부분 **검증·보강·엣지케이스**에 관한 것이다. 처음부터 새로 짜는 게 아니다.

## 2. 절차 A — 포인트 조회 (PRD 9.4)

### A-1. 데이터 흐름 확인
`app/points/page.tsx`는 서버 컴포넌트에서 로그인 사용자의 `point_events`를 조회하고
`sumPoints()`로 총점을 계산해 보여준다.

### A-2. RLS 확인 (보안 핵심)
`point_events`는 **본인 행만** SELECT되어야 한다. `supabase/migrations/0002_rls.sql`에서
`user_id = auth.uid()` 정책이 걸려 있는지 확인한다. 이게 없으면 남의 포인트가 보인다.

### A-3. 체크리스트
- [ ] 로그인 안 한 상태로 `/points` 접근 → `requireUser()`가 로그인으로 보내는지
- [ ] 이벤트 0건일 때 "내역 없음" 표시
- [ ] 총점이 이벤트 합계와 일치 (음수 차감 이벤트가 있어도 정확한지)
- [ ] 마이페이지(`app/mypage`)에서도 총 포인트가 동일하게 보이는지

## 3. 절차 B — 추천인 코드 (PRD 9.5)

### B-1. 코드 발급
- 회원가입 트리거(`0003_triggers.sql`)에서 `referral_codes`가 자동 생성된다.
- 트리거 이전 계정 대비 방어: `app/referral/page.tsx`의 `getOrCreateCode()`가
  코드가 없으면 서버에서 즉석 발급한다. (self-heal 패턴)

### B-2. 추천 링크 규격
```
/signup?ref=추천코드
```
가입 폼은 `ref` 쿼리를 읽어 숨은 값으로 넣고, 로그인 직후
`POST /api/auth/after-signup`에 `{ refCode }`로 전달한다.

### B-3. 추천 보상 지급 로직 (핵심 = 중복 방지)
`lib/after-signup.ts`의 규칙을 그대로 유지한다:
1. `refCode`로 `referral_codes`에서 추천인(`referrer_id`) 조회 (활성 코드만)
2. **자기 자신 추천 금지**: `referrerId !== userId`
3. `referrals`에 insert — `referred_id`가 **UNIQUE**라 이미 추천받은 사람이면 insert 실패
   → 이 실패를 **정상 흐름으로 처리**해서 보상을 skip (중복 보상 차단)
4. 성공 시에만 가입자 `referral_join` + 추천인 `referral_inviter` 지급

### B-4. 멱등성 (두 번 눌러도 안전)
`runAfterSignup`은 맨 앞에서 "이미 `signup` 이벤트가 있으면 전체 no-op". 로그인 직후
클라이언트가 실수로 두 번 호출해도 포인트가 두 번 들어가지 않는다.

### B-5. 체크리스트
- [ ] 잘못된/비활성 코드로 가입 → 추천 없이 신규가입 포인트만
- [ ] 같은 유저가 after-signup 2회 호출 → 포인트 1회만
- [ ] A가 B를 추천, 이후 B가 C 추천링크로 재가입 시도 → `referred_id` UNIQUE로 차단
- [ ] 추천 성공 시 A(추천인)와 B(가입자) **양쪽 다** 지급됐는지

## 4. 절차 C — 관리자 포인트 정책 (PRD 9.3)

### C-1. 권한 가드 (가장 중요)
정책 수정은 **admin만** 가능해야 한다. `app/api/admin/policies/route.ts`는
`assertAdminRequest(serverDb)`로 role을 검증하고, 통과 시에만 **service_role 클라이언트**로
업데이트한다. (RLS 우회는 서버에서 admin 확인 후에만)

- 실패 응답 규격: 미로그인 `401`, 비관리자 `403`, 잘못된 입력 `400`

### C-2. 수정 가능한 필드
`parsePolicyPatch()`가 허용하는 것만 통과시킨다 (화이트리스트):
- `points` (정수)
- `is_active` (boolean)
- `name` (정책명)
- `policy_key`는 **식별자라 변경 대상이 아님** (WHERE 조건으로만 사용)

### C-3. "하드코딩 금지" 원칙 지키기
새 포인트 지급 지점을 추가할 때는 **항상** `grantPoints(db, { policyKey })`를 거친다.
숫자를 코드에 직접 쓰지 않는다. 새 정책이 필요하면:
1. `point_policies`에 새 `policy_key` 행 추가 (마이그레이션 or 시드)
2. 지급 지점에서 `grantPoints`로 해당 키 호출
3. 관리자 화면에 자동으로 노출됨 (정책 목록을 테이블에서 읽으므로)

### C-4. 체크리스트
- [ ] 일반 회원이 PATCH 직접 호출 → 403
- [ ] `points`를 0으로 → 이후 지급이 0점으로 적립되는지
- [ ] `is_active=false`로 끄면 → `grantPoints`가 `null` 반환하고 미지급
- [ ] `updated_at`이 갱신되는지

## 5. 상용화 확장 (MVP 이후)

- **포인트 사용(차감)**: 상담 결제에 포인트 사용 → 음수 `points`로 `point_events` 적립,
  `sumPoints`는 그대로 동작. 단, **잔액 부족 검증**을 지급 전에 트랜잭션으로 추가해야 함.
- **동시성**: 동시에 여러 지급이 몰릴 때 이중지급 방지 → DB 유니크 제약 또는
  `select ... for update`/RPC(Postgres 함수) 안에서 처리 권장.
- **정책 이력**: 정책값 변경 이력이 필요하면 `point_policy_history` 테이블 추가.
- **관리자 수동 지급/회수 화면**: 특정 회원에게 임의 포인트 지급/차감 UI.
