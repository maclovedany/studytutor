# PRD — 전문가 1:1 코칭 상담 플랫폼 “코치링”

## 1. 제품 개요

### 1.1 서비스명

코치링 Coachring

### 1.2 서비스 한 줄 설명

전문가와 사용자를 연결하여 15분 무료 상담을 제공하고, 이후 결제를 통해 유료 상담으로 전환하는 1:1 코칭 상담 플랫폼

### 1.3 서비스 목적

사용자가 전문가 상담을 부담 없이 시작할 수 있도록 무료 상담 기회를 제공하고, 상담 만족도가 생긴 시점에 자연스럽게 유료 결제로 전환시키는 구조를 만든다.

동시에 추천인 코드, 포인트, 자동 메시지, 문자인증, 관리자 설정 기능을 통해 서비스 운영자가 회원 유입과 전환을 관리할 수 있도록 한다.

---

## 2. 문제 정의

### 2.1 사용자의 문제

- 전문가 상담을 받고 싶지만 처음부터 결제하기에는 부담이 있음
- 어떤 전문가에게 상담을 받아야 할지 판단하기 어려움
- 상담 전 신뢰를 형성할 기회가 부족함
- 상담 신청 후 진행 상황이나 혜택을 쉽게 확인하기 어려움

### 2.2 운영자의 문제

- 신규 회원 유입을 위한 추천인/포인트 정책을 유연하게 운영하기 어려움
- 가입 후 사용자를 다시 방문하게 만드는 자동 메시지 시스템이 없음
- 상담 전환, 결제 유도, 유료회원 전환 과정을 수동으로 관리해야 함
- 앱 확장, 광고 수익화, 회원 인증 구조를 처음부터 어떻게 설계해야 할지 어려움

## 3. 타깃 고객

### 3.1 1차 타깃

전문가의 도움을 받고 싶지만 처음부터 유료 결제하기 부담스러운 사용자

예시:

- 이직 준비 직장인
- 커리어 방향을 고민하는 사회초년생
- PT, 학습, 재테크, 창업 등 특정 분야의 코칭이 필요한 사람
- 전문가 상담을 처음 이용해보는 사용자

### 3.2 관리자/운영자

코칭 서비스를 운영하는 개인 또는 소규모 사업자

예시:

- 커리어 코치
- 운동 코치
- 학습 코치
- 재테크 코치
- 1:1 상담 기반 서비스를 만들고 싶은 창업자

## 8. 주요 사용자 흐름

### 8.1 신규 가입 흐름

1. 사용자가 서비스에 접속한다.
2. 로그인 또는 회원가입을 진행한다.
3. 추천인 코드가 있는 경우 추천인 정보를 저장한다.
4. 신규 회원 프로필이 생성된다.
5. 신규가입 포인트를 지급한다.
6. 추천인 코드로 가입한 경우 추천인과 가입자에게 각각 포인트를 지급한다.
7. 가입 후 1일차/3일차/7일차 메시지 예약을 생성한다.
8. 사용자는 마이페이지에서 포인트와 회원 상태를 확인한다.

### 8.2 추천인 포인트 흐름

1. 기존 회원이 자신의 추천인 코드를 확인한다.
2. 추천 링크를 복사해서 공유한다.
3. 신규 사용자가 추천 링크로 접속한다.
4. 신규 사용자가 가입한다.
5. 가입자에게 추천가입 포인트를 지급한다.
6. 추천인에게 추천성공 포인트를 지급한다.
7. 모든 지급 내역은 포인트 이벤트로 저장된다.

### 8.3 휴대폰 인증 흐름

1. 사용자가 휴대폰 번호를 입력한다.
2. 서버가 6자리 인증번호를 생성한다.
3. 인증번호와 만료시간을 저장한다.
4. 개발 단계에서는 인증번호를 화면 또는 콘솔에 표시한다.
5. 사용자가 인증번호를 입력한다.
6. 서버가 인증번호와 만료시간을 검증한다.
7. 성공 시 `phone_verified_at`을 저장한다.
8. 인증 완료 회원만 상담예약을 진행할 수 있다.

### 8.4 상담/결제 전환 흐름

1. 인증 완료 회원이 상담예약을 진행한다.
2. 상담 시작 버튼을 누른다.
3. Zoom 링크를 통해 상담을 시작한다.
4. 서비스 화면에서는 상담 시작 시간을 저장한다.
5. 15분 타이머가 작동한다.
6. 15분이 지나면 결제 유도 모달을 표시한다.
7. 사용자가 데모 결제 버튼을 누른다.
8. 회원의 `tier`를 `paid`로 변경한다.
9. 유료회원은 상담 계속하기 버튼을 사용할 수 있다.
10. 유료회원에게는 결제 유도 버튼을 숨긴다.

## 9.2 회원 프로필 기능

### 기능 설명

로그인 사용자의 서비스 회원 정보를 관리한다.

### 요구사항

- 로그인 사용자마다 `profiles` 데이터가 있어야 한다.
- `role` 값을 통해 관리자와 일반 사용자를 구분한다.
- `tier` 값을 통해 무료회원과 유료회원을 구분한다.
- 마이페이지에서 이메일, 이름, role, tier를 보여준다.

### 기본값

| 필드 | 기본값 |
| ---- | ------ |
| role | user   |
| tier | free   |

## 9.3 포인트 정책 기능

### 기능 설명

관리자가 포인트 지급 정책을 수정할 수 있다.

### 요구사항

- 포인트 수치는 코드에 하드코딩하지 않는다.
- `point_policies` 테이블에서 포인트 값을 읽는다.
- 관리자는 정책명, 포인트 수치, 활성화 여부를 수정할 수 있다.
- 정책이 비활성화되어 있으면 포인트를 지급하지 않는다.

### 정책 예시

| policy_key            |             이름 | 포인트 |
| --------------------- | ---------------: | -----: |
| signup                |         신규가입 |   1000 |
| referral_join         | 추천 가입자 보상 |   2000 |
| referral_inviter      |      추천인 보상 |   3000 |
| consultation_reserved |    상담예약 보상 |    500 |
| payment_completed     |    결제완료 보상 |   5000 |

---

## 9.4 포인트 이벤트 기능

### 기능 설명

실제 포인트 지급/차감 내역을 기록한다.

### 요구사항

- 포인트 잔액만 저장하지 않는다.
- 모든 적립/차감 내역을 `point_events`에 기록한다.
- 마이페이지에서 총 포인트와 포인트 내역을 보여준다.
- 총 포인트는 `point_events`의 합계로 계산한다.

### 포인트 이벤트 예시

| 이벤트   | 설명                                      |
| -------- | ----------------------------------------- |
| 신규가입 | 회원가입 시 포인트 지급                   |
| 추천가입 | 추천 코드로 가입한 사용자에게 포인트 지급 |
| 추천성공 | 추천한 사람에게 포인트 지급               |
| 상담예약 | 상담예약 완료 시 포인트 지급              |
| 결제완료 | 결제 성공 시 포인트 지급                  |

## 9.5 추천인 코드 기능

### 기능 설명

회원이 추천인 코드를 통해 다른 사용자를 초대할 수 있다.

### 요구사항

- 회원마다 추천인 코드가 생성된다.
- 추천 링크는 `/signup?ref=추천코드` 형식으로 사용한다.
- 추천 코드로 가입한 경우 추천인 정보를 저장한다.
- 가입자와 추천인에게 각각 다른 포인트를 지급할 수 있다.
- 중복 추천 보상을 방지해야 한다.

---

## 9.6 카카오 로그인 기능

### 기능 설명

카카오 로그인을 통해 사용자가 쉽게 가입/로그인할 수 있도록 한다.

### 요구사항

- 카카오 로그인 버튼을 제공한다.
- 로그인 성공 시 Supabase Auth 사용자가 생성된다.
- 신규 사용자라면 `profiles` 데이터를 생성한다.
- 로그인 후 마이페이지로 이동한다.
- 카카오 Redirect URI와 Supabase Redirect URL 설정이 필요하다.

### MVP 처리

- 강의에서는 구조 이해와 가능 범위 내 구현/시연을 목표로 한다.
- Redirect URI 문제로 수업 시간이 지연될 경우, 준비된 시연 계정으로 흐름을 설명한다.

## 9.7 자동 메시지 예약 기능

### 기능 설명

회원가입 후 1일차, 3일차, 7일차 메시지를 예약한다.

### 요구사항

- 신규 회원 생성 시 메시지 예약 3건을 생성한다.
- 메시지 발송 채널은 `kakao`로 저장한다.
- 실제 카카오톡 발송은 MVP에서는 구현하지 않는다.
- 관리자 페이지에서 예약된 메시지를 확인할 수 있다.
- 메시지는 `pending`, `sent`, `failed` 상태를 가진다.

### 메시지 예시

| 시점  | 메시지                                            |
| ----- | ------------------------------------------------- |
| 1일차 | 회원가입해주셔서 감사합니다                       |
| 3일차 | 아직 상담을 예약하지 않으셨다면 지금 시작해보세요 |
| 7일차 | 첫 상담 혜택이 곧 종료됩니다                      |

---

## 9.8 휴대폰 인증 기능

### 기능 설명

상담예약 전 사용자의 휴대폰 번호를 인증한다.

### 요구사항

- 사용자가 휴대폰 번호를 입력할 수 있다.
- 서버가 6자리 인증번호를 생성한다.
- 인증번호는 5분 뒤 만료된다.
- 개발 단계에서는 실제 SMS를 보내지 않고 인증번호를 화면 또는 콘솔에 표시한다.
- 사용자가 인증번호를 입력하면 서버에서 검증한다.
- 성공 시 `profiles.phone_verified_at`을 저장한다.
- 인증 완료된 회원만 상담예약을 할 수 있다.

### 상용화 시 확장

- Solapi 또는 유사 SMS API를 서버에서 호출한다.
- API Secret은 클라이언트에 두지 않고 서버 환경변수로 관리한다.

## 9.9 상담예약 기능

### 기능 설명

사용자가 1:1 상담을 예약하고 Zoom 링크를 통해 상담을 시작할 수 있다.

### 요구사항

- 인증 완료 회원만 상담예약 가능하다.
- 상담예약 시 `consultations` 데이터가 생성된다.
- 상담에는 상태값이 있어야 한다.
- 상담 시작 버튼을 누르면 `started_at`을 저장한다.
- Zoom 링크를 연결한다.

### 상담 상태

| 상태             | 의미      |
| ---------------- | --------- |
| reserved         | 예약됨    |
| started          | 상담 시작 |
| payment_required | 결제 필요 |
| paid             | 결제 완료 |
| resumed          | 상담 재개 |
| completed        | 상담 종료 |

## 9.10 15분 무료 상담 타이머 기능

### 기능 설명

상담 시작 후 15분이 지나면 결제 유도 화면을 표시한다.

### 요구사항

- 상담 시작 시간을 기준으로 타이머를 계산한다.
- 15분 이전에는 무료 상담 상태로 표시한다.
- 15분 이후에는 결제 유도 모달을 표시한다.
- 무료회원에게만 결제 유도 모달을 표시한다.
- 유료회원은 결제 유도 없이 상담을 계속할 수 있다.

## 9.11 데모 결제 기능

### 기능 설명

실제 결제 연동 전, 결제 성공 흐름을 데모로 구현한다.

### 요구사항

- 결제 유도 모달에 데모 결제 버튼을 제공한다.
- 데모 결제 버튼 클릭 시 결제 성공으로 처리한다.
- 사용자의 `tier`를 `paid`로 변경한다.
- 상담 상태를 `paid` 또는 `resumed`로 변경한다.
- 유료회원에게는 결제 버튼을 숨긴다.

### 상용화 시 확장

- 토스페이먼츠, Stripe 등 결제사 연동
- 서버에서 결제 승인 검증
- 결제 내역 저장
- 환불/취소 처리

### MVP 범위

## 9.13 AdMob 광고 구조

### 기능 설명

앱 수익화를 위한 AdMob 광고 구조를 이해한다.

### 요구사항

- AdMob 계정 생성이 필요하다.
- 앱 등록이 필요하다.
- 광고 단위 ID를 발급받아야 한다.
- 테스트 광고 ID로 먼저 테스트해야 한다.
- Expo Go만으로는 제한이 있을 수 있으며 Development Build 또는 EAS Build가 필요할 수 있다.
- 실제 광고 적용은 Google 정책을 준수해야 한다.

### MVP 범위

- 강의에서는 구조 설명과 적용 단계 이해까지 진행한다.
- 실제 광고 수익화는 후속 과제로 진행한다.

## 13. 비기능 요구사항

### 13.1 보안

- Service Role Key는 클라이언트에 노출하지 않는다.
- SMS/Kakao API Secret은 서버 환경변수에 저장한다.
- 관리자 기능은 admin role만 접근 가능해야 한다.
- 포인트 지급은 서버에서 처리한다.
- 사용자는 자신의 포인트 내역만 조회할 수 있어야 한다.

### 13.2 확장성

- 포인트 정책은 DB 기반으로 운영자가 수정 가능해야 한다.
- 메시지 예약은 실제 발송 API와 분리되어야 한다.
- 웹과 앱은 같은 DB/API를 사용할 수 있어야 한다.
- 실제 결제 연동으로 확장 가능해야 한다.

### 13.3 사용성

- 사용자는 현재 포인트를 쉽게 확인할 수 있어야 한다.
- 추천인 코드는 복사하기 쉬워야 한다.
- 상담 예약 버튼은 인증 상태에 따라 활성/비활성 처리되어야 한다.
- 결제 유도는 15분 이후 명확하게 표시되어야 한다.

## 15. 성공 기준

### 15.1 MVP 성공 기준

- 사용자가 가입하고 마이페이지를 볼 수 있다.
- 사용자가 추천인 코드를 확인하고 공유할 수 있다.
- 추천 코드 가입 시 포인트가 지급된다.
- 사용자가 포인트 내역을 조회할 수 있다.
- 관리자가 포인트 정책을 수정할 수 있다.
- 가입 시 1/3/7일 메시지 예약이 생성된다.
- 사용자가 휴대폰 인증을 완료할 수 있다.
- 인증된 사용자만 상담예약을 할 수 있다.
- 상담 시작 후 15분 타이머가 작동한다.
- 15분 이후 결제 유도 화면이 표시된다.
- 데모 결제 후 유료회원으로 전환된다.
- Expo 앱에서 포인트를 조회할 수 있다.

### 15.2 학습 성공 기준

- 수강생이 서비스 기획을 기능 단위로 쪼갤 수 있다.
- 수강생이 Claude Code에게 작은 단위로 작업을 요청할 수 있다.
- 수강생이 포인트 정책과 포인트 이벤트의 차이를 설명할 수 있다.
- 수강생이 카카오 로그인과 자동 회원가입 구조를 이해한다.
- 수강생이 SMS Secret을 클라이언트에 두면 안 되는 이유를 설명할 수 있다.
- 수강생이 웹과 앱이 같은 DB/API를 공유한다는 구조를 이해한다.
