# 카카오 로그인 + 자동 메시지(예약 발송) + 문자인증 — 구현 절차

> 대상 범위: PRD 9.6(카카오 로그인) / 9.7(자동 메시지 예약) / 9.8(휴대폰 인증)
> 오늘의 핵심 목표: **카톡 알람을 회원에게 1일·3일·7일·10분 등 원하는 시점에 자동 발송**
> 스택: Next.js 16, Supabase(Auth/DB/Cron), 발송사 = Solapi(솔라피, 알림톡+SMS 통합)

## 0. 오늘의 핵심 개념 — "예약 발송"은 어떻게 동작하는가

가장 먼저 이해할 것: **"예약 발송"은 두 개의 분리된 일**이다. 이걸 헷갈리면 전체 설계가 꼬인다.

```
[1] 예약 만들기            [2] 실제 쏘기 (스케줄러)
회원가입/이벤트 발생  →  message_jobs에      →  주기적으로(예: 1분마다)      →  Solapi로 즉시 발송
                        "언제 뭘 보낼지" 저장    "지금 보낼 때가 된 것" 픽업        status: sent/failed 기록
                        (scheduled_at, pending)  (scheduled_at <= now)
```

- **[1] 예약 만들기**는 이미 구현됨 → `lib/after-signup.ts`가 가입 시 1/3/7일치 `message_jobs`를
  `pending` 상태로 3건 생성한다.
- **[2] 실제 쏘기**가 오늘 만들 부분 → 지금은 아무도 이 `pending` 행을 발송하지 않는다(=MVP 스텁).

### 왜 "발송사에 바로 예약"하지 않고 자체 큐를 두는가?

두 가지 방식이 있다:

| 방식 | 설명 | 장점 | 단점 |
| --- | --- | --- | --- |
| **A. 발송사 예약 위임** | 가입 시점에 Solapi에 `scheduledDate=7일뒤`로 미래 예약 접수 | 서버 크론 불필요 | 취소/수정 어렵고, "10분 뒤" 같은 상대 이벤트·조건부 발송이 유연치 못함, 상태 동기화 별도 |
| **B. 자체 큐 + 폴링(권장)** | `message_jobs`에 쌓고, 스케줄러가 due 픽업 후 **즉시발송** | 취소·조건 분기·재시도·로그 완전 제어, 10분/1일 무엇이든 동일 코드 | 스케줄러(크론) 1개 필요 |

**이 프로젝트는 B를 택한다.** 이유: `message_jobs` 테이블이 이미 `pending/sent/failed` +
`scheduled_at` + `sent_at`으로 딱 이 패턴에 맞게 설계돼 있다. "10분 뒤 상담 리마인드"처럼
상대시간·조건부 발송도 같은 파이프라인으로 처리된다.

### "1일·3일·7일·10분"의 정체

전부 **기준 시각 + 오프셋**일 뿐이다. 예약을 만들 때 `scheduled_at = 기준시각 + 오프셋`으로 계산한다.

- 가입 시각 + 1일 / +3일 / +7일 → 온보딩 리마인드
- 상담 시작 시각 + 10분 → "무료 15분이 곧 끝나요" 결제 유도 알림
- 결제 시각 + 10분 → 영수증/후기 요청

즉 `day` 단위든 `minute` 단위든 **오프셋 숫자만 다르고 구조는 동일**하다.

## 1. 사전 준비물 (계정·발송 자격)

카카오 관련 기능은 **외부 사업자 심사·등록**이 필요해서 코드보다 행정 절차가 먼저다.

### 1-1. 카카오 로그인
1. [Kakao Developers](https://developers.kakao.com) 앱 생성
2. **REST API 키** 발급
3. **Redirect URI** 등록: 로컬 `http://localhost:3000/auth/kakao/callback`, 운영 도메인도 추가
4. 카카오 로그인 활성화 + 동의항목(닉네임/이메일 등) 설정
5. (선택) **카카오싱크**: 간편 회원가입까지 붙이려면 비즈앱 전환 + 검수 필요

### 1-2. 알림톡(자동 메시지) — Solapi 경유
> 알림톡 = 카카오톡으로 오는 **정보성 메시지**(광고 아님). 친구추가 불필요, 24시간 발송 가능.
1. [Solapi](https://solapi.com) 가입, **API Key / API Secret** 발급
2. **카카오 비즈니스 채널** 생성 후 Solapi에 연동
3. **알림톡 템플릿** 등록 → 카카오 **검수 승인**(영업일 며칠 소요). 승인된 템플릿만 발송 가능
   - 변수는 `#{이름}`, `#{날짜}` 같은 **치환 문구**로 작성
4. 광고성/자유 문구가 필요하면 알림톡 대신 **친구톡**(채널 친구 대상)

### 1-3. 문자인증(SMS) — Solapi 경유
1. Solapi에서 **발신번호 사전 등록**(통신사 규정, 등록된 번호로만 발송 가능)
2. SMS(90byte)면 충분 — 인증번호는 짧다

### 1-4. 환경변수 (`.env.local`, 서버 전용 — 절대 클라이언트 노출 금지)
```bash
# 카카오 로그인
KAKAO_REST_API_KEY=
KAKAO_CLIENT_SECRET=            # 사용 설정 시
NEXT_PUBLIC_KAKAO_REDIRECT_URI=http://localhost:3000/auth/kakao/callback

# Solapi (알림톡 + SMS 공용)
SOLAPI_API_KEY=
SOLAPI_API_SECRET=
SOLAPI_SENDER_PHONE=           # 사전 등록한 발신번호
SOLAPI_PFID=                   # 카카오 채널(플러스친구) ID — 알림톡용
```
> `.env.local`은 이미 gitignore. Secret은 **서버 라우트/Edge Function에서만** 읽는다.

## 2. 절차 A — 카카오 로그인 (PRD 9.6)

> ✅ **확정: 이 프로젝트는 A-1(Supabase Auth의 Kakao Provider)로 간다.**
> A-2(REST 직접 구현)는 채택하지 않으며, 아래엔 비교용 참고로만 남겨둔다.

### A-1. (채택) Supabase Auth의 Kakao Provider 사용
Supabase가 OAuth 왕복을 대신 처리해줘서 코드가 가장 적다.
1. Supabase 대시보드 → Authentication → Providers → **Kakao** 활성화
2. 카카오 REST API 키/Secret 입력, Supabase가 주는 **Callback URL**을 카카오 Redirect URI에 등록
3. 로그인 버튼:
   ```ts
   await supabase.auth.signInWithOAuth({
     provider: "kakao",
     options: { redirectTo: `${SITE_URL}/auth/callback` },
   });
   ```
4. 콜백 처리 후 **기존 `after-signup` 흐름 재사용**: 프로필 보장(`ensureProfile`) →
   `POST /api/auth/after-signup`로 신규가입/추천 포인트 + 1/3/7일 메시지 예약까지 그대로 탄다.
   → **카카오로 가입해도 포인트/추천/메시지 예약이 자동 연결**된다.

### A-2. (미채택 · 참고용) 순수 REST API 직접 구현
Supabase Provider를 안 쓸 때. 카카오 로그인 공식 흐름:
1. 인가코드 요청: `https://kauth.kakao.com/oauth/authorize?client_id=...&redirect_uri=...&response_type=code`
2. 사용자 동의 → Redirect URI로 `code` 수신
3. 서버에서 `code` → 토큰 교환: `POST https://kauth.kakao.com/oauth/token`
4. 토큰으로 사용자 정보: `GET https://kapi.kakao.com/v2/user/me`
5. 이메일/카카오ID로 **기존 회원 조회 → 신규면 가입, 있으면 연동**
> REST 방식은 토큰·세션 관리를 직접 해야 하므로 Supabase Auth를 쓰는 이 프로젝트에선 비권장.

### A-3. 체크리스트
- [ ] 로컬/운영 Redirect URI **양쪽** 등록 (누락 시 KOE006 에러)
- [ ] 카카오 최초 로그인 사용자도 `profiles`/`referral_codes` 자동 생성되는지
- [ ] 카카오 가입자도 1/3/7일 `message_jobs`가 생기는지

## 3. 절차 B — 자동 메시지 = 예약 발송 시스템 (PRD 9.7) ★오늘의 본론★

### B-1. [1단계] 예약 만들기 — 이미 됨 (+ 확장)
`lib/after-signup.ts`가 가입 시 1/3/7일 `message_jobs`를 `pending`으로 생성 중.
**"10분" 같은 새 시점**을 추가하려면 오프셋을 분 단위로 넣으면 된다:
```ts
// 예: 상담 시작 10분 후 결제유도 알림 예약
scheduled_at: new Date(startedAt.getTime() + 10 * 60 * 1000).toISOString()
```
- 템플릿은 `template_key`로 식별 (예: `welcome_d1`, `remind_d3`, `expire_d7`, `pay_10min`)
- 실제 알림톡 발송 시 이 `template_key`를 **Solapi 승인 템플릿**과 매핑한다

### B-2. [2단계] 발송 워커 — 오늘 새로 만들 것
"지금 보낼 때가 된" pending 행을 픽업해 Solapi로 쏘는 함수. **엔드포인트로 노출**한다.

- 파일 제안: `app/api/jobs/dispatch-messages/route.ts` (POST)
- 로직:
  ```
  1) message_jobs에서 status='pending' AND scheduled_at <= now() 조회 (배치 100건)
  2) 각 건마다 Solapi 발송:
       - channel='kakao' → 알림톡 (pfId + templateId + 치환변수)
       - channel='sms'   → 문자 (fallback)
  3) 성공 → status='sent', sent_at=now()  /  실패 → status='failed' (+ 에러 로그)
  4) 동시성 방지: 픽업 시 'sending'으로 선점(UPDATE ... RETURNING) 후 발송  ← 중복발송 차단
  ```
- **보안**: 이 엔드포인트는 아무나 못 부르게 `CRON_SECRET` 헤더 검증 (스케줄러만 호출).

Solapi Node SDK 발송 예시(알림톡):
```ts
import { SolapiMessageService } from "solapi";
const svc = new SolapiMessageService(process.env.SOLAPI_API_KEY!, process.env.SOLAPI_API_SECRET!);
await svc.send({
  to: user.phone,
  from: process.env.SOLAPI_SENDER_PHONE!,
  kakaoOptions: {
    pfId: process.env.SOLAPI_PFID!,
    templateId: "승인된_템플릿_ID",
    variables: { "#{이름}": user.display_name },
  },
});
```

### B-3. [3단계] 스케줄러 — 워커를 주기적으로 부르기
셋 중 하나. **Supabase Cron(pg_cron) 권장**(별도 서버·유료플랜 불필요, DB에 내장).

| 옵션 | 방법 | 비고 |
| --- | --- | --- |
| **Supabase Cron (권장)** | 대시보드 Cron에서 "HTTP Request" 잡 생성 → 위 dispatch 엔드포인트를 1분마다 호출 | 모든 플랜에서 pg_cron 기본 제공, `select cron.schedule(...)` |
| Supabase Edge Function 스케줄 | Edge Function에 발송 로직을 넣고 cron으로 예약 | 실행시간 여유 큼 |
| Vercel Cron | `vercel.json`에 크론 등록 | Hobby는 **하루 1회 제한** → 1일/3일/7일엔 되지만 **10분 단위는 불가**. 무료로 잦게 돌리려면 cron-job.org(무료)로 엔드포인트 핑 |

> "10분 간격" 요구가 있으므로 **하루 1회짜리 Vercel Hobby 크론은 부적합**.
> Supabase Cron으로 1분(또는 1~5분)마다 dispatch를 호출하는 구성을 기본으로 한다.

pg_cron 등록 예시(1분마다 dispatch 호출):
```sql
select cron.schedule(
  'dispatch-messages',
  '* * * * *',
  $$ select net.http_post(
       url    := 'https://<도메인>/api/jobs/dispatch-messages',
       headers:= '{"x-cron-secret":"<CRON_SECRET>"}'::jsonb
     ) $$
);
```

### B-4. 관리자 조회 (이미 화면 존재)
`app/admin/messages/page.tsx`에서 예약 목록/상태(pending/sent/failed)를 확인.
발송 워커 붙인 뒤에는 `sent_at`·실패 사유까지 보이도록 컬럼 보강 권장.

### B-5. 체크리스트
- [ ] `scheduled_at`이 과거인 pending을 dispatch가 실제로 픽업하는가
- [ ] 같은 잡이 2번 안 쏴지는가 (선점 UPDATE로 동시성 차단)
- [ ] 발송 실패 시 `failed`로 남고 재시도 정책이 있는가 (예: 3회까지)
- [ ] 미인증/전화번호 없는 회원은 skip 또는 SMS fallback 하는가
- [ ] `CRON_SECRET` 없이 dispatch 직접 호출 → 401

## 4. 절차 C — 휴대폰 문자인증 (PRD 9.8)

### C-1. 코어 로직 — 이미 됨
`lib/phone.ts`: `genCode()`(6자리), `expiryFrom()`(+5분), `verifyCode()`(used/expired/mismatch/ok).
API 라우트도 존재: `app/api/phone/send/route.ts`, `app/api/phone/verify/route.ts`.

### C-2. 발송(send) 흐름
1. 사용자가 휴대폰 번호 입력 → `POST /api/phone/send`
2. 서버: `genCode()` → `phone_verifications`에 `{phone, code, expires_at}` 저장
3. **개발 단계**: 실제 SMS 대신 **콘솔/응답에 코드 노출**(PRD 명시) — 지금 상태
4. **상용화**: 여기서 Solapi SMS 호출로 교체
   ```ts
   await svc.send({ to: phone, from: process.env.SOLAPI_SENDER_PHONE!, text: `인증번호: ${code}` });
   ```
   → **코드 값은 응답 바디에서 제거**(콘솔/응답 노출 금지)

### C-3. 검증(verify) 흐름
1. 사용자가 코드 입력 → `POST /api/phone/verify`
2. `verifyCode(record, input, now)`로 판정 → 성공 시 `profiles.phone_verified_at` 저장
3. **인증 완료자만 상담예약 가능** (PRD 9.9) — 상담예약 라우트에서 `phone_verified_at` 확인

### C-4. 보안·남용 방지 (상용화 필수)
- [ ] 발송 **rate limit** (번호·IP당 분/시간당 횟수 제한) — SMS 폭탄/과금 방어
- [ ] 인증번호 **재발송 시 이전 코드 무효화**
- [ ] 시도 횟수 제한(예: 5회 실패 시 잠금)
- [ ] API Secret은 서버 환경변수만 (C-1 라우트는 서버 전용 유지)
- [ ] 인증번호는 로그/응답에 남기지 않기(상용화)

### C-5. 체크리스트
- [ ] 만료(5분 초과) 코드 → `expired`
- [ ] 이미 쓴 코드 재사용 → `used`
- [ ] 틀린 코드 → `mismatch`
- [ ] 성공 시 `phone_verified_at` 기록되고 상담예약 게이트 통과

## 5. 오늘 실제 작업 순서 요약

1. **행정 준비**: 카카오 앱/채널 생성, Solapi 가입·발신번호 등록·알림톡 템플릿 검수 신청
   (승인은 시간이 걸리니 **가장 먼저** 신청) → 환경변수 채우기
2. **카카오 로그인**: Supabase Kakao Provider 켜고 로그인 버튼 + 콜백에서 `after-signup` 연결 (§2 A-1)
3. **발송 워커 신설**: `app/api/jobs/dispatch-messages` — pending 픽업 → Solapi 발송 → 상태 갱신 (§3 B-2)
4. **스케줄러 등록**: Supabase Cron에서 1분마다 dispatch 호출 + `CRON_SECRET` 검증 (§3 B-3)
5. **10분 등 새 시점 예약 추가**: 상담 시작/결제 기준 오프셋으로 `message_jobs` 생성 (§3 B-1)
6. **SMS 실발송 교체**: `phone/send`의 스텁을 Solapi SMS로, 응답에서 코드 제거 (§4 C-2)
7. **검증**: 예약 → 크론 → 발송 → `sent` 상태 변화를 로컬(콘솔 목업)부터 end-to-end 확인

> MVP 데모 팁: 발송사 검수 전이라면 `dispatch-messages`가 실제 Solapi 대신
> **콘솔 로그로 "발송했다 치고" status만 sent로** 바꾸게 해두면, 스케줄러/큐 파이프라인
> 전체를 실계정 없이도 시연할 수 있다. 실계정 승인 후 발송 함수만 교체하면 끝.
